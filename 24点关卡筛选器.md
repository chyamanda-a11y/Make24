实现一个可落地的 24 点关卡筛选器。

目标不是“筛出有解题”，而是筛出适合放进 108 关主线的题。
核心要求是：主解可信、教学标签可信、章节节奏可信、全局不重复。

## 一、筛选器职责

筛选器必须同时解决下面 4 类问题：

1. 解存在，但人工填写的 `intendedSolution` 不是最自然主解。
2. 同一组数字被重复放进主线，尤其是跨章节重复。
3. 题本身有解，但它的真实主解不符合目标 `phaseId` 的教学目标。
4. 单题看起来没问题，但放进章节后体感重复或节奏失衡。

筛选器对“新生成题”和“现有题库”一视同仁。
不能因为题已经写进 json，就跳过重复检查或主解检查。

## 二、输入与输出

输入建议至少包含：

```ts
type LevelCandidate = {
  id?: number;
  chapterId?: number;
  phaseId: string;
  numbers: [number, number, number, number];
  target: 24;
  intendedSolution?: string;
  keyIdea?: string;
  teachingTags?: string[];
  difficulty?: number;
};
```

输出建议包含：

```ts
type FilterResult = {
  accepted: boolean;
  numbersKey: string;
  allSolutionCount: number;
  dominantSolution?: string;
  runnerUpSolution?: string | null;
  dominantKeyIdea?: string;
  dominantTags?: string[];
  dominantDifficulty?: number;
  structureFamily?: string;
  rhythmFingerprint?: string;
  scoreBreakdown?: Record<string, number | boolean | string>;
  rejectReasons: string[];
};
```

`rejectReasons` 必须是结构化原因列表，不能只输出一句自然语言。

## 三、核心术语

### 1. numbersKey

同一组数字忽略顺序后的唯一 key：

```ts
const numbersKey = [...numbers].sort((a, b) => a - b).join(",");
```

主线 108 关内，`numbersKey` 必须全局唯一。
不是“章节内唯一”，也不是“相邻关卡唯一”，而是整条主线唯一。

### 2. allSolutions

必须枚举当前数字的所有合法表达式解。
不能直接相信人工填写的 `intendedSolution`。

要求：

1. 四个数字必须全部且仅使用一次。
2. 只允许 `+ - * /` 和括号。
3. 除零非法。
4. 需要去掉纯文本重复解。
5. 如后续成本允许，最好再做一层“结构等价去重”。

### 3. dominantSolution

`dominantSolution` 指这组数字里，最自然、最容易被玩家优先想到的解。

`answerExpression`、`intendedSolution`、`keyIdea`、`teachingTags`、`difficulty` 都不能直接人工指定，
必须从 `dominantSolution` 反推。

### 4. runnerUpSolution

`runnerUpSolution` 指仅次于主解的候选解。
它的作用不是展示给玩家，而是帮助判断：

1. 当前主解是否稳定。
2. 是否存在“明显更简单的替代解”。
3. 这题是否适合拿去教某个特定思路。

## 四、筛选流程

建议固定为下面顺序：

1. 生成 `numbersKey`。
2. 枚举 `allSolutions`。
3. 对每个解计算可读性/自然度评分。
4. 选出 `dominantSolution` 与 `runnerUpSolution`。
5. 从 `dominantSolution` 推断 `dominantKeyIdea / dominantTags / dominantDifficulty / structureFamily`。
6. 检查是否匹配目标 `phaseId`。
7. 检查是否与全局已选题重复。
8. 检查章节内节奏冲突。
9. 通过则产出标准化关卡元数据；失败则输出完整 `rejectReasons`。

## 五、dominantSolution 选取规则

### 1. 基本原则

主解必须尽量接近“真人第一反应”，而不是“数学上也成立的一条解”。

例如 `2,2,6,6`：

- `((2*6)+(2*6))`
- `((2+6)*(6/2))`

两者都合法，但前者通常更直观。
因此不能把后者硬标成“这题主思路就是先做 8 再乘 3”。

### 2. 排序优先级

建议用“分层比较”而不是单一权重总分。
按下面顺序逐层比较，前一层优先级永远高于后一层：

1. 更少分数中间值
2. 更少负数中间值
3. 更少除法
4. 更少反直觉结构
5. 更小的最大中间值
6. 更少结构失衡
7. 更贴近常见友好锚点
8. 更容易口头描述

其中：

- 反直觉结构包括：`a/(b/c)`、`(a-b)` 后再拿去当分母、先绕出 1 再恢复原值、明显的恒等变换
- 常见友好锚点包括：`24 / 12 / 8 / 6 / 4 / 3 / 2 / 1`
- “更容易口头描述”指玩家能自然说出“先做 12 再加 12”“先做 8 再乘 3”

### 3. 必须记录的解特征

每个候选解至少要统计：

```ts
type SolutionFeatures = {
  usesDivision: boolean;
  usesSubtraction: boolean;
  fractionCount: number;
  negativeCount: number;
  maxAbsIntermediate: number;
  anchorHits: number;
  structureDepth: number;
  imbalance: number;
  hasTrivialIdentityPattern: boolean;
  hasCounterIntuitiveDivision: boolean;
};
```

### 4. 明显更简单替代解

如果存在另一个合法解，在前 4 个高优先级维度上严格优于当前目标解，
并且不在更高优先级维度上更差，则视为“明显更简单替代解存在”。

一旦发生这种情况：

1. 不能再用较难那条解去定义 `keyIdea`
2. 不能再按较难思路给 `difficulty`
3. 如果更简单主解与目标 phase 冲突，则整题淘汰

建议使用明确拒绝码：

- `SIMPLER_ALTERNATIVE_EXISTS`

## 六、从 dominantSolution 反推教学标签

筛选器不能信任人工填写的：

- `keyIdea`
- `teachingTags`
- `difficulty`

这些字段必须从主解反推。

建议至少反推出：

```ts
type DominantSemantics = {
  dominantKeyIdea: string;
  dominantTags: string[];
  dominantDifficulty: number;
  structureFamily:
    | "double-same-merge"
    | "make-12-then-double"
    | "make-8-then-times-3"
    | "make-6-then-times-4"
    | "make-1-then-scale"
    | "divide-then-combine"
    | "order-sensitive"
    | "fraction-driven"
    | "hidden-anchor"
    | "other";
};
```

`structureFamily` 必须比 `keyIdea` 更稳定，因为它要用于节奏去重。

## 七、phase 适配规则

不能只检查这题“有解”，还要检查“真实主解是否适合该 phase”。

建议按 `phaseId` 定义硬约束与软目标。

### novice-a

硬约束：

1. `dominantSolution` 不能用除法
2. 不能出现分数中间值
3. 不能依赖顺序敏感结构

软目标：

1. 一眼可见 `12+12`、`8*3`、`6*4`
2. 难度应偏低

### novice-b

硬约束：

1. 不能出现分数中间值
2. 主思路应围绕明显中间值

软目标：

1. `dominantKeyIdea` 优先是“凑 12 / 凑 8 / 凑 6”
2. 两步思考可以开始出现，但不能太绕

### novice-c

硬约束：

1. 可以有简单整除
2. 不能依赖丑分数

软目标：

1. 允许“先除出 2 / 3 / 4”
2. 允许轻度顺序意识

### advanced-a

硬约束：

1. 主解应明显比新手章多一步规划
2. 不能只是新手题的重复皮肤

软目标：

1. 两步思考常态化
2. 可以有除法或减法，但不是为了恶心人

### advanced-b

硬约束：

1. 主解应体现顺序重要性
2. 先做哪一步会显著影响结果

### advanced-c

硬约束：

1. 主解应包含隐藏中间值或轻误导
2. 不能只是显眼的直接凑数

### challenge-a / challenge-b / challenge-c

原则：

1. 难度可上升，但仍要公平
2. 只允许心算友好的分数
3. 不能用“唯一解很丑”来制造挑战

如果 `dominantSolution` 与目标 phase 的硬约束冲突，直接淘汰：

- `DOMINANT_SOLUTION_MISMATCH_PHASE`

## 八、全局去重规则

### 1. numbersKey 全局唯一

整个 108 关主线内，同一 `numbersKey` 只能出现一次。

明确拒绝码：

- `DUPLICATE_NUMBERS_KEY`

### 2. 不允许“旧配置豁免”

如果现有题库里已经有重复 `numbersKey`，
筛选器必须照样报错或淘汰，不能因为它已经写进 json 就自动保留。

### 3. 相似题不等于重复题

以下情况不是同一题，但仍要做节奏去重：

1. 数字不同，但 `dominantKeyIdea` 相同
2. 数字不同，但 `structureFamily` 相同
3. 数字不同，但玩家第一感受高度相似

## 九、章节节奏去重规则

除了 `numbersKey` 全局唯一，还要做章节内的“体感去重”。

建议构造：

```ts
const rhythmFingerprint = [
  phaseId,
  structureFamily,
  dominantKeyIdea,
  usesDivision ? "div" : "no-div",
  hasFraction ? "frac" : "no-frac",
  anchorBucket,
].join("|");
```

建议最少做以下限制：

1. 相同 `dominantKeyIdea` 不能连续出现超过 2 关
2. 相同 `structureFamily` 在任意 6 关窗口内不能过多重复
3. 分数型题不能连续出现
4. 同一种“先做 1 / 先做 3 / 先凑 8”不要短窗口内反复出现

明确拒绝码：

- `RHYTHM_DUPLICATE`
- `KEY_IDEA_RUN_TOO_LONG`
- `STRUCTURE_FAMILY_OVERUSED`

## 十、rejectReasons 设计

`rejectReasons` 不能为空数组时，说明该题被淘汰。

建议至少支持下面这些拒绝码：

```ts
type RejectReason =
  | "NO_LEGAL_SOLUTION"
  | "DUPLICATE_NUMBERS_KEY"
  | "SIMPLER_ALTERNATIVE_EXISTS"
  | "DOMINANT_SOLUTION_MISMATCH_PHASE"
  | "RHYTHM_DUPLICATE"
  | "KEY_IDEA_RUN_TOO_LONG"
  | "STRUCTURE_FAMILY_OVERUSED"
  | "INVALID_METADATA"
  | "UNSTABLE_DOMINANT_SOLUTION";
```

同时输出辅助信息：

1. `duplicateWith`
2. `runnerUpSolution`
3. `scoreGap`
4. `phaseCheckResult`
5. `rhythmWindowSnapshot`

## 十一、落地建议

现有生成/校验脚本至少需要补下面能力：

1. 对所有合法解做完整枚举
2. 计算 `dominantSolution`
3. 用 `dominantSolution` 回填 `answerExpression / keyIdea / teachingTags / difficulty`
4. 对整个 108 关做 `numbersKey` 全局去重
5. 对章节窗口做 `rhythmFingerprint` 去重
6. 输出结构化 `rejectReasons`

## 十二、最低验收标准

筛选器算合格，至少要满足：

1. 能识别 `2,2,6,6` 这类“存在更直观主解”的题
2. 能识别跨章节重复数字组
3. 能阻止用更难副解去硬贴教学标签
4. 能给出可读的淘汰原因
5. 能让人工快速看懂“为什么保留这题 / 为什么淘汰这题”

如果做不到这 5 条，说明它还只是“有解检查器”，还不是“主线关卡筛选器”。
